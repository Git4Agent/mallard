# Plan: Global `~/.agent-sync` Directory (Non-Invasive App Files)

Status: **IMPLEMENTED** (2026-07-12) — `Roots` remap, `machine.json`
registry, in-root exclusion + cleanup, overlap validation, and the
unit/integration tests below are all in `lib.rs` / `harness.rs`.
Date: 2026-07-12

## 1. Goal

Stop writing app-generated files inside the synced agent roots. Today the
only such files are the plugin locks:

```text
~/.codex/agent-sync/codex-plugins.lock.json
~/.claude/agent-sync/claude-plugins.lock.json
```

All app-generated per-root records move to one global, app-owned directory,
`~/.agent-sync/`. After this change the agent roots contain only what Codex
and Claude themselves produce: the app reads them and syncs them but never
plants its own files there. This removes a real class of risk — agents
cleaning or checksumming unknown files in their own trees, users
`git init`-ing a root, dotfile managers and backups picking up our
metadata.

## 2. Load-bearing decision: remap physical location, keep logical paths

Everything in the sync engine speaks **logical paths**: cloud manifests,
baselines, the tier allowlist, merge-driver dispatch, and conflict-copy
naming all key on `.codex/agent-sync/codex-plugins.lock.json`. Keep those
unchanged. Change only the local mount: logical `.codex/agent-sync/**` now
lives physically at `~/.agent-sync/codex/**` instead of
`<codex_root>/agent-sync/**` (same for `.claude`).

Consequences, all for free:

- No allowlist, merge-driver, baseline, manifest, or cloud-layout change.
  Existing cloud profiles work unchanged.
- Cross-version convergence: an old build materializes the lock inside its
  root, a new build in `~/.agent-sync`. The Tier 2 drivers are
  byte-deterministic on content, so mixed-version machines still converge.
- Every path consumer that routes through `Roots::abs`/`Roots::rel`
  (apply, backups, conflict copies, status, editor validation) inherits the
  remap automatically.

Alternatives rejected:

- A profile-metadata object outside the file namespace: new CAS/validation
  surface for one file, and it breaks "pull reads exactly what the
  manifest references".
- A synced `~/.agent-sync` root/profile: re-opens the third-root decision
  (PLAN_PORTABLE_AGENT_SETUP.md §4.2) for no gain — this directory holds
  per-machine state plus files that already sync under existing roots.

## 3. Layout

```text
~/.agent-sync/
  machine.json          # machine-local registry; never syncs (by construction)
  codex/                # physical home of logical .codex/agent-sync/**
    codex-plugins.lock.json
  claude/               # physical home of logical .claude/agent-sync/**
    claude-plugins.lock.json
```

Subdirectories use non-dotted slugs (`codex`, `claude`) so they cannot be
mistaken for real agent roots by other tooling.

## 4. `machine.json` — per-profile local path registry

```json
{
  "schema": 1,
  "roots": {
    ".codex":  { "local_path": "/Users/hequ/.codex",  "profile_id": "01H...A9" },
    ".claude": { "local_path": "/Users/hequ/.claude", "profile_id": "01H...B4" }
  }
}
```

- Regenerated by the app on config save, push, and pull.
- Records, per logical root: the physical root path on this machine and
  (informationally) the currently linked profile id for the active remote.
- `sync_config.json` in app data remains the single configuration
  authority. `machine.json` is a self-describing mirror for humans and
  tooling; the app writes it and never reads it back as config. No second
  source of truth.
- It contains machine-local absolute paths and must never sync. This is
  guaranteed structurally, not by an allowlist rule: it sits at the top of
  `~/.agent-sync/`, outside both remapped subtrees, so `Roots::rel` maps it
  to no logical path and it cannot enter a manifest at all.

## 5. Stale in-root `agent-sync/` — no migration, exclude and remove

There is no content migration. The lock is recaptured from local machine
state before every push, so an old in-root lock carries nothing worth
preserving; cloud copies are untouched (the logical path is unchanged).

Policy:

- **Never read** `<root>/agent-sync/` again. The root scan must skip that
  subtree explicitly. This is mandatory, not hygiene: after the remap a
  stale in-root file would otherwise be walked as the same logical path as
  the remapped file — two physical files claiming one logical
  `.{root}/agent-sync/...` entry.
- **Remove best-effort.** On capture/push, if `<root>/agent-sync/` exists,
  delete the known app-generated filenames (the plugin lock and its
  `*.sync-conflict-*` siblings) and remove the directory if it is then
  empty. Unknown files are left in place and the directory is kept — the
  app never deletes data it did not generate.
- Pull writes only to the new home (`Roots::abs` points there).

## 6. Safety rules

- Root overrides may not be `~/.agent-sync` or nest it either direction;
  extend the existing root-overlap validation.
- Only exact allowlisted lock files sync; neither the logical
  `agent-sync/` prefix nor the physical directories are ever allowlisted
  wholesale (unchanged rule from the plugin locks).
- `machine.json` stays outside the remapped subtrees permanently; nothing
  with absolute paths may be placed inside `codex/` or `claude/`.
- `validate_editable_path` accepts the remapped area exactly like root
  paths; everything else stays rejected.

## 7. Code touchpoints

### `src-tauri/src/lib.rs`

- `agent_sync_dir(home)` helper → `<home>/.agent-sync`, using the same
  `$HOME` resolution as the roots so the test harness home-swap keeps
  working.
- `Roots::abs`: prefix exception `.codex/agent-sync/` →
  `<agent_sync>/codex/` and `.claude/agent-sync/` →
  `<agent_sync>/claude/`, checked before the normal root join.
- `Roots::rel`: inverse mapping, checked before the root-prefix checks.
- `collect_upload_files`: additionally walk `~/.agent-sync/codex` and
  `~/.agent-sync/claude`, emitting logical
  `.codex/agent-sync/<name>` / `.claude/agent-sync/<name>` entries;
  existing eligibility filtering applies unchanged.
- Root scans skip `<root>/agent-sync/` entirely (§5 — prevents a stale
  in-root file from colliding with the remapped logical path).
- Best-effort stale-dir cleanup per §5, invoked from capture/push.
- Root-overlap validation per §6.
- `machine.json` writer invoked from config save, push, and pull.

### `src-tauri/src/codex_plugins.rs`

- Route lock read/write paths through `Roots` (audit for any direct
  `home.join(LOCK_REL)`); the old in-root location is never read.
- No schema or merge changes.

## 8. Tests

Unit:

- `Roots::abs`/`rel` round-trip for the remapped prefix on default and
  custom root paths.
- `~/.agent-sync/machine.json` maps to no logical path (`rel` → `None`).
- Overlap validation rejects a root set to or nested with `~/.agent-sync`.

Dual-backend integration (existing harness):

- Push captures the lock into `~/.agent-sync/codex/` and syncs it; the
  agent root gains no `agent-sync/` directory.
- Fresh-machine pull materializes the lock under `~/.agent-sync/`, leaving
  the pristine root untouched.
- A stale in-root lock is ignored by the scan (no logical-path collision
  with the remapped file) and removed on the next push; an unknown file in
  `<root>/agent-sync/` survives and keeps the directory alive.
- Mixed layout converges: one side simulates an old build writing in-root,
  both sides end with identical lock bytes at the canonical logical path.

## 9. Non-goals

- Moving app data (`sync_config.json`, baselines, backups) into
  `~/.agent-sync`. It already lives outside the agent roots; relocating it
  is churn without a non-invasiveness gain. Revisit only with a concrete
  need.
- Changing the cloud layout, lock schemas, or merge semantics.
- Syncing `machine.json` or any absolute path.

## 10. Relationship to PLAN_PORTABLE_AGENT_SETUP

Decided (PLAN_PORTABLE_AGENT_SETUP_V2.md D8): there is no setup lock in
Phases 1–2 — readiness derives from the synced raw files, the plugin
locks, and conflict-copy siblings. If Follow-up A ever adds one, it
inherits this layout automatically (logical
`.{root}/agent-sync/setup.lock.json`, physical
`~/.agent-sync/{codex,claude}/`). The portable-setup plan also specifies a
second top-level machine-local file here, `~/.agent-sync/local-state.json`
(readiness memory: reviewed-hook hashes, dismissed issues) — same
unsyncable-by-construction placement as `machine.json`. This plan stands
on today's plugin locks alone.
